# 0.1.3 设计需求

## 背景

当前版本在 Codex 等 MCP 客户端中使用时存在两个主要体验问题：

1. 每次开启新对话都会启动一个新的 Chrome 实例，即使本次对话完全没有使用 Chrome DevTools MCP。
2. AI 操作浏览器时，Chrome 窗口有时会弹出或抢占焦点，打断用户正在进行的其他工作。

0.1.3 的目标是降低 Chrome DevTools MCP 的侵入感：只有真正需要浏览器能力时才启动 Chrome，并尽量让有界面浏览器在后台静默启动。

## 目标

- 默认不在 MCP server 启动时打开 Chrome。
- 在第一次真实调用浏览器相关 MCP 能力时再启动 Chrome。
- 保持 MCP 初始化、握手、工具发现流程正常可用。
- 默认使用兼容性更好的 headful Chrome，但尽量避免弹窗和抢焦点。
- 保留旧版 eager 启动行为，方便用户显式回退。
- 保留 headless 模式，满足完全无窗口场景。
- 不引入第三方运行时依赖。
- 不破坏已有 profile mode、随机端口、临时 profile、清理逻辑。

## 非目标

- 不自行实现或硬编码 chrome-devtools-mcp 的工具列表。
- 不保证 Windows 或 Chrome 在所有情况下绝不抢焦点；quiet 模式是 best-effort。
- 不改变下游 MCP server 的协议语义。
- 不自动复用用户已有 Chrome 主进程。
- 不默认复制 cookies、session、登录数据等敏感 profile 数据。

## 用户体验需求

### Lazy 启动

默认行为应为 lazy：

```powershell
uvx chrome-devtools-mcp-canpoint -- npx -y chrome-devtools-mcp@latest --browser-url={browser_url}
```

在该模式下：

- MCP 客户端启动 server 时，不应立即启动 Chrome。
- `initialize`、`notifications/initialized`、`tools/list` 等初始化和发现请求应正常转发给下游 MCP。
- 第一次出现真实浏览器使用请求时，wrapper 应先启动 Chrome，等待 DevTools endpoint 可用，再转发原始请求。
- 对当前 Chrome DevTools MCP，关键触发请求是 `tools/call`。
- 为未来兼容，也应将 `resources/read`、`prompts/get` 视为 lazy 启动触发点。

### Eager 回退

提供显式参数保留旧行为：

```powershell
uvx chrome-devtools-mcp-canpoint --launch-mode eager -- npx -y chrome-devtools-mcp@latest --browser-url={browser_url}
```

在 eager 模式下：

- wrapper 启动后立即 materialize profile。
- 启动 Chrome 并等待 DevTools endpoint 可用。
- 再启动或桥接下游 MCP 请求。
- 行为应尽量接近 0.1.2。

### Quiet 默认窗口模式

默认窗口模式应为 quiet：

```powershell
uvx chrome-devtools-mcp-canpoint --window-mode quiet -- npx -y chrome-devtools-mcp@latest --browser-url={browser_url}
```

quiet 模式要求：

- 保持 headful Chrome，以兼容登录、页面渲染、扩展和某些反自动化场景。
- Chrome 参数包含 `--start-minimized`。
- Windows 下使用 `STARTUPINFO` 尝试以 minimized/no-activate 状态启动。
- 文档明确说明 quiet 模式只能减少抢焦点，不能保证完全静默。

### Visible 和 Headless 模式

提供显式窗口模式：

```powershell
uvx chrome-devtools-mcp-canpoint --window-mode visible -- npx -y chrome-devtools-mcp@latest --browser-url={browser_url}
uvx chrome-devtools-mcp-canpoint --window-mode headless -- npx -y chrome-devtools-mcp@latest --browser-url={browser_url}
```

要求：

- `visible`：不添加 quiet/headless 特有参数，适合调试。
- `headless`：添加 `--headless=new`，完全无窗口。
- `--headless` 保留为向后兼容别名，等价于 `--window-mode headless`。

## CLI 设计

新增参数：

```text
--launch-mode lazy|eager
--window-mode quiet|visible|headless
```

默认值：

```text
--launch-mode lazy
--window-mode quiet
```

保留参数：

```text
--headless
```

兼容规则：

- 如果用户传入 `--headless`，实际窗口模式应视为 `headless`。
- 如果用户传入 `--launch-mode eager`，应保留启动即开 Chrome 的旧体验。
- 如果用户未传任何新参数，应获得 lazy + quiet 的新默认体验。

## 架构设计

### ProfilePlan

新增 profile planning 概念，将“选择路径”和“实际物化 profile”拆开。

启动 MCP server 时只做 planning：

- 选择随机 port。
- 确定 `CHROME_DEVTOOLS_URL` / `BROWSER_URL`。
- 确定未来要使用的 `user_data_dir` 路径。
- 将这些信息传给下游 MCP。

真正启动 Chrome 前才 materialize profile：

- `isolated`：创建临时 profile 目录。
- `inherit`：验证源 profile 是否存在。
- `copy`：复制源 profile，并应用已有安全过滤规则。

这样可以避免“新对话但未使用 Chrome”时创建无意义 profile 或复制数据。

### ChromeSessionManager

新增 session manager 管理 Chrome 生命周期：

- `ensure_started()` 幂等启动 Chrome。
- 第一次调用时 materialize profile、解析 Chrome 路径、启动 Chrome、等待 DevTools。
- 后续调用如果 Chrome 仍在运行，直接返回。
- `cleanup()` 在 Chrome 从未启动时必须是安全空操作。
- cleanup 只终止 wrapper 自己启动的 Chrome 进程。
- 只有实际 materialize 的临时 profile 才会被删除。

### MCP stdio lazy proxy

替换 client stdin 到 downstream stdin 的简单字节桥接，改为轻量 MCP stdio proxy：

- 按 `Content-Length` 读取 MCP frame。
- 解析 JSON-RPC payload，检查 `method`。
- 非触发请求原样转发。
- 触发请求先调用 `ChromeSessionManager.ensure_started()`。
- Chrome 启动成功后原样转发请求。
- Chrome 启动失败且请求带 `id` 时，返回 JSON-RPC error，不转发该请求。
- 下游 stdout 到 client stdout 仍保持简单字节桥接。

## 错误处理需求

- lazy 启动 Chrome 失败时，不应让 MCP 客户端无限等待。
- 对带 `id` 的请求返回 JSON-RPC error：
  - `code`: `-32000`
  - `message`: 包含 Chrome startup failed 和异常信息。
- 对 notification 类无 `id` 消息，记录 stderr 后继续处理后续消息。
- cleanup 必须可重复调用，不因 Chrome 未启动而报错。

## 清理需求

- lazy 模式未触发 Chrome 时，不启动 Chrome，不创建临时 profile，不删除未创建目录。
- Chrome 启动失败时，终止已启动的 Chrome 进程并清理已 materialize 的临时 profile。
- 正常退出、stdin 关闭、Ctrl+C、SIGTERM 时，沿用已有清理逻辑。
- Windows 下继续使用 Job Object 约束 wrapper 创建的 Chrome 进程树。

## 测试需求

新增或覆盖以下测试：

- `plan_user_data_dir` 在 copy 模式下不立即复制 profile。
- `materialize_profile` 才实际复制 profile。
- quiet 模式添加 `--start-minimized`。
- visible 模式不添加 quiet/headless 参数。
- headless 模式添加 `--headless=new`。
- `ChromeSessionManager.ensure_started()` 幂等。
- `ChromeSessionManager.cleanup()` 在未启动 Chrome 时不终止进程、不删除目录。
- lazy bridge 对 `initialize`、`tools/list` 不启动 Chrome。
- lazy bridge 对 `tools/call` 先启动 Chrome 再转发。
- lazy 启动失败时返回 JSON-RPC error。
- `main()` 默认 lazy，不预启动 Chrome。
- `main()` 在 eager 模式下先启动 Chrome 再运行下游 MCP。

## 文档需求

README 需要说明：

- 默认已经从 eager 改为 lazy。
- lazy 的触发时机。
- 如何使用 `--launch-mode eager` 回退旧行为。
- quiet 是默认窗口模式，但只是 best-effort。
- 如何使用 `--window-mode visible` 调试。
- 如何使用 `--window-mode headless` 或 `--headless` 完全隐藏窗口。

## 验收标准

- 新对话只初始化 MCP、不调用浏览器工具时，不启动 Chrome 实例。
- 第一次调用 Chrome DevTools MCP 工具时，Chrome 被启动并可连接 DevTools endpoint。
- quiet 模式下 Chrome 尽量最小化/不抢焦点启动。
- `--launch-mode eager` 可恢复旧版启动行为。
- `--headless` 仍可使用。
- 单元测试全部通过。
- 包构建可成功产出 sdist 和 wheel。
